# ImmiSign SaaS Portal: UX Gap and Live Demo Readiness Report

This report presents a thorough **User Experience (UX) Gap Analysis** of the ImmiSign platform. Rather than analyzing code structures, this document adopts the perspective of a **Registered Migration Agent (RMA)** conducting a live demo of the software. 

Our goal is to identify all placeholder data, mock behaviors, hardcoded strings, empty states, and incomplete forms that an agent or visa client would notice in active use.

## 2. Granular Page-by-Page UX Audit

### 🏠 Dashboard Home Page (`/workspace/[agency]/dashboard`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Placeholder Greeting**: The greeting *"Good day, Rajwant"* is a hardcoded string, rather than greeting the logged-in user profile name dynamically.
  - **Mock Metric Cards**: The executive statistics (Total Sent: 24, Awaiting: 12, Signed: 18, This month: $42k) are hardcoded mocks that do not query live agreement database tables.
  - **Mock Activity Feeds**: All dashboard notifications (*"Harpreet signed Partner Visa agreement"*, *"Invoice INV-048 was paid"*) are pre-baked, static items that do not represent active workspace logs.
  - **Missing Controls**: No buttons are available to dismiss individual notification items, filter the activity logs, or customize metric metrics widgets.

---

### 📝 Agreements Workspace & Wizard (`/workspace/[agency]/agreements`)
* **UX Classification**: `Needs Polish`
* **Identified UX Gaps**:
  - **Pre-Baked Client List**: The main workspace directory displays 11 static demo agreements (Harpreet, Gurpreet, etc.). Real agreements generated by the user appear temporarily but do not persist once sessions clear.
  - **Hardcoded Preview Summaries**: Opening the agreement detail card displays a premium visual PDF summary, but the legal scope of work text is hardcoded to New South Wales (NSW) OMARA parameters.
  - **Missing Signature Placements**: The 6-step Agreement Wizard (`/agreements/new`) collects client, matter, and GST fee details, but does not provide an interface to drag-and-drop signature seals or date input fields onto the PDF draft.
  - **Mock Actions**: Action buttons in the dropdown (*"Void Agreement"*, *"Send Email Reminder"*) trigger visual toast alerts but do not affect database states or dispatch notifications.

---

### 📂 Application Approvals (`/workspace/[agency]/approvals`)
* **UX Classification**: `Production Ready`
* **Identified UX Gaps**:
  - **Dev Mode Email Bypass**: Clicking *"Request Changes"* appends comments and transitions the workflow status (Draft -> Changes Requested), but does not dispatch real email alerts to client inboxes because SMTP is bypassed in Developer Mode.
  - **Visual Placeholder Details**: Seed data defaults to static client subclass records, though new applications uploaded by agents populate dynamic tables.

---

### 📤 Send Document (`/workspace/[agency]/documents/send`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Mock Signatures**: Dropping a PDF and clicking *"Send"* successfully shows a loader and prompts *"Document sent for signature"*, but does not execute a real API handshake with SignWell to establish secure signing bounds.
  - **Static Practitioner Pick**: The RMA dropdown only displays preset demo practitioners rather than fetching active members from the agency's team settings.
  - **Empty Upload Warnings**: The dropzone has no file-type validation to prevent users from dropping broken image files, text files, or empty assets.

---

### 🗃️ Document Library (`/workspace/[agency]/documents`)
* **UX Classification**: `Needs Polish`
* **Identified UX Gaps**:
  - **Static Directory Structure**: Displays default mock folders (Incoming Visas, Retainer Agreements) that are not linked to dynamic folders in the Supabase private storage bucket.
  - **Missing Directory Tools**: No search bar or category tag filter is implemented inside the documents table.
  - **Hardcoded File Metrics**: File sizes (e.g. "2.4 MB") and upload timestamps are hardcoded strings.

---

### 📑 Templates Library (`/workspace/[agency]/templates`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Static Template List**: Displays 4 pre-baked standard templates. You cannot create new templates or delete existing ones.
  - **No Template Editor**: Clicking a template instantly imports preset fields into the agreement wizard, but there is no template editor to modify the boilerplate text, pricing configurations, or default clauses.

---

### 👥 Clients Directory (`/workspace/[agency]/clients`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Static List & Metrics**: The client listing page displays 4 hardcoded client names. The matters count and total financial values (e.g., "$8,700") are hardcoded strings.
  - **No Dynamic Profile Merges**: Clicking on a client displays a static matter timeline that is not dynamically compiled from active agreements, PDF uploads, or approval events.
  - **Incomplete Forms**: The *"Create Client"* drawer collects basic details but lacks advanced OMARA fields (such as dual passport numbers, family sponsor links, or bridge visa expiration timelines).

---

### 📊 Practice Analytics & Reports (`/workspace/[agency]/analytics`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Mock Visualization Curves**: The glowing SVG bezier analytics grid and conversion graphs represent static coordinate maps rather than live database entries.
  - **Missing Date Filters**: The graphs have no controls to select custom ranges (YTD, last 30 days) or compare performance across different practitioners.
  - **Static Financial Logs**: Reports display hardcoded invoice totals and audit stamps. There are no export buttons (such as PDF download or CSV/Excel exports).

---

### ⚙️ Settings Workspace (`/workspace/[agency]/settings`)
* **UX Classification**: `Needs Polish`
* **Identified UX Gaps**:
  - **Local Session Memory**: Updating branding elements (custom color picks and company initials) instantly renders on screen, but is stored in local session memory rather than synced permanently back to the Supabase database.
  - **Mock Security logs**: The active user session security tables display simulated IP addresses and device names.
  - **Static Default Settings**: Matter types and payment milestones are static outlines.

---

### 💳 SaaS Billing & Subscriptions (`/workspace/[agency]/billing`)
* **UX Classification**: `Mocked`
* **Identified UX Gaps**:
  - **Hardcoded Stripe Details**: Shows Pro Enterprise subscription billing information ($129/mo) and a mock VISA card ending in 4242.
  - **Mock Seat Provisions**: Entering a target seat count and clicking *"Confirm & Charge"* plays a dynamic progress bar and logs updates to the workspace dashboard, but does not bill a live credit card on Stripe.

## 3. High-Priority "Live Demo" Warnings for RMAs

If presenting a live demonstration of ImmiSign to a Registered Migration Agent (RMA) or potential investor, be aware of these functional limitations:

1. **Autosave and Session Resets**: Because client data and workspace settings use Zustand stores and local mocks, **refreshing the browser tab will reset settings changes**, custom color selections, newly added boilerplate clauses, and recently invited team members.
2. **Signature Simulation**: Sending an agreement initiates the signing flow and shows the *Sent* status in the agreements table, but since the sandbox SignWell keys are bypassed, you cannot complete an actual e-signature cycle on a document.
3. **No Active Email Alerts**: Adding comments to dynamic application reviews or inviting practitioners will not dispatch real inbox notification emails.
4. **Billing Upgrades Bypassed**: The Stripe customer portal is fully mocked; clicking billing updates redirects to custom success widgets rather than loading an active Stripe-hosted interface.
